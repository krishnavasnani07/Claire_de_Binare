# LR-041 Evidence: Chaos — Redis/Postgres Restart Recovery

- Issue: `#787`
- Status: `IMPLEMENTED`
- Last updated: `2026-03-10`

## 1. Scope

LR-041 requires that the system recovers gracefully from Redis and Postgres restarts within defined time thresholds, without data loss or execution leakage.

Pass criteria (Issue #787):
- Redis Restart: Recovery <30 seconds
- Postgres Restart: Recovery <60 seconds
- No real execution during or after recovery
- Shadow runtime mode preserved throughout

## 2. Implementation Anchors

- `scripts/drills/lr041_redis_postgres_failure_runner.py`
  - deterministic drill runner using `docker restart`
  - measures container health recovery + downstream service recovery
  - enforces fail-closed invariants: shadow mode, zero execution (LR-030)
  - collateral restarts (dependent services via `restart: unless-stopped`) recorded as observation, not failure
  - produces machine-readable evidence artifacts (timeline, summary, verdict)

- `tests/chaos/test_lr041_redis_postgres.py`
  - chaos gate test gated by `RUN_CHAOS_TESTS=1`
  - runs redis_restart scenario, verifies artifacts and invariants
  - marked `@pytest.mark.chaos` + `@pytest.mark.local_only`
