# LR-040 Evidence: 72h Soak Test

- Issue: `#786`
- Control: `LR-040`
- Status: `IMPLEMENTED`
- Last updated: `2026-03-20`

## 1. Scope

LR-040 validates that the system runs stably for 72 hours without memory leaks,
crashes, or performance degradation.

This control is a separate 72h verdict/evidence path. It does not define the
committed P5 core artifact contract and does not, by itself, replace a
committed artifact set under `reports/p5_canary/<YYYY-MM-DD>/`.

Pass criteria (Issue #786):
- 72h without restart/crash (zero restart policy)
- Memory usage stable (<10% growth across any service)
- CPU usage <70% average
- No critical errors in logs

## 2. Implementation Anchors

- `infrastructure/scripts/lr040_soak_gate_eval.py`
  - evaluates `soak_monitor.sh` artifacts against #786 pass criteria
  - fail-closed: missing artifacts, insufficient duration, or threshold violations ‚Üí FAIL
  - outputs `lr040_soak_gate_eval.json` with machine-readable verdict

- `infrastructure/scripts/soak_monitor.sh`
  - hourly monitoring automation (Issue #428)
  - checks: container restarts, service health, resource snapshots (6h), DB growth (12h), disk space
  - creates `soak_test_FAILED.txt` marker on abort condition

- `infrastructure/compose/soak.yml`
  - Docker Compose overlay with resource limits for soak runs

- `infrastructure/monitoring/alerts.yml`
  - Prometheus alert group `soak_test_gates` with abort/investigate classification

- `infrastructure/monitoring/grafana/dashboards/claire_soak_test_v1.json`
  - "Claire - 72h Soak Test Monitor" dashboard

- `docs/operations/72H_SOAK_TEST_RUNBOOK.md`
  - operational procedure for manual 72h soak execution

## 3. Gate Evaluator Checks

The evaluator (`lr040_soak_gate_eval.py`) enforces:

| Check | Criterion | Fail-closed |
|---|---|---|
| `hourly_log_present` | `hourly_checks.log` exists | yes |
| `duration_gte_72h` | first-to-last timestamp span ‚â• 72h | yes |
| `no_failed_marker` | `soak_test_FAILED.txt` absent | yes |
| `no_restart_alerts` | `restart_alerts.log` empty or absent | yes |
| `memory_growth_below_10pct` | max service memory growth < 10% | yes |
| `cpu_avg_below_70pct` | overall CPU average < 70% | yes |
| `resource_snapshots_present` | ‚â• 2 resource snapshots exist | yes |

`soak_test_*` directories are canonical LR-040 evidence. `soak_validation_*`
directories are short verification runs that test the same monitor mechanics
but are not LR-040 evidence. The gate evaluator returns `NOT_APPLICABLE` for
validation runs (Issue #1278).

These directories are raw run artifacts. They are not committed P5 core
artifacts.

The dedicated LR-040 verdict anchor is
`artifacts/soak_test_YYYYMMDD_HHMMSS/lr040_soak_gate_eval.json`.
If P5 governance later needs to reference LR-040 in committed form, the
normative committed path is separate:
`reports/p5_canary/<YYYY-MM-DD>/lr040/lr040_soak_gate_eval.json`.

## 5. Status Interpretation

- `IMPLEMENTED` is justified by: gate evaluator with fail-closed checks,
  monitoring script, runbook, alerts, and dashboard ‚Äî all tooling needed to
  execute and evaluate a 72h soak.
- `PASS` requires an actual 72h soak run with
  `lr040_soak_gate_eval.json` verdict `PASS`. This document does not claim that.
- A `PASS` verdict on LR-040 is a necessary 72h evidence anchor only. It does
  not, by itself, create a committed P5 core artifact set and does not move P5
  out of `NO-GO`.
