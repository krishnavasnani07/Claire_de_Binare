# Context Intelligence — Agent Memory Authoring and Import Policy v1

**Status**: Design (Wave-14 Follow-up)
**Authority**: Issue #2571 / Follow-up Arc of PR #2565
**Parent**: Epic #1976
**Schema Reference**: `infrastructure/surrealdb/context_intelligence_v0.surql` (`agent_memory` table)
**Memory Model Reference**: `docs/surrealdb/scoped-agent-memory-model-v1.md`
**Guardrail**: No trading state, no secrets, no Live-Go, no Echtgeld-Go, no runtime change.

---

## 1. Purpose and Scope

`agent_memory` is intentionally not generated by the repo indexer (`context_indexer.py`).
Agent memory is runtime/agent-authored context, not a deterministic repo-scan artefact.

The `scoped-agent-memory-model-v1.md` document defines the memory model (types, TTL,
supersede rules, write constraints). This document extends that model by defining:

- Who and what is allowed to create `agent_memory` records
- How `agent_memories.jsonl` records are validated before JSONL import
- Allowed and prohibited sources
- Stale and supersede handling at import time

**Scope:**
- Memory authoring policy (who/what creates records)
- JSONL import validation rules
- Stale and supersede handling at import
- Minimal safe import slice

**Non-Goals:**
- No modification of `scoped-agent-memory-model-v1.md` (that document remains authoritative
  for the memory model itself)
- No automated memory generation from repo scan
- No hidden runtime writes
- No DB writes without explicit future scope
- No Live-Readiness or Echtgeld claims from memory
- No schema changes
- No `scoped-agent-memory-model-v1.md` content duplication

## 2. Authoring Policy

### 2.1 Allowed Authoring Paths

| Path | Allowed | Conditions |
|---|---|---|
| Human manual authoring (JSONL file) | ✅ Yes | No additional constraint beyond field requirements |
| Agent in-session authoring → explicit JSONL export step | ✅ Yes | Export step requires Human-GO; agent MUST NOT self-import |
| Automated repo scanner | ❌ No | Memory is not a deterministic scan artefact |
| Automatic export from session logs | ❌ No | Session logs are ephemeral context, not evidence-backed memory |
| Backfill from git history | ❌ No | Git history is evidence source, not memory content |

### 2.2 Agent-Authored Memory

When an agent authors memory records during a session:
1. Records MUST be written to a local JSONL export file, not directly to the DB.
2. A human MUST review the records before they are submitted for import.
3. The agent MUST NOT set `human_go: true` on any memory-adjacent field.
4. The agent's `created_by` field MUST be set to the agent's canonical identifier.

---

## 3. Mandatory Fields for Import

Every `agent_memory` record in `agent_memories.jsonl` MUST satisfy all constraints defined
in `scoped-agent-memory-model-v1.md` Section 4, and additionally the following import-specific
rules:

| Field | Requirement | Import Rule |
|---|---|---|
| `memory_id` | Mandatory | Stable deterministic ID; MUST be unique within the batch |
| `scope` | Mandatory | MUST NOT be empty, `global`, or `*` |
| `namespace` | Mandatory | MUST NOT be empty |
| `memory_type` | Mandatory | One of the 6 canonical types (see `scoped-agent-memory-model-v1.md` Section 3) |
| `content` | Mandatory | MUST NOT be empty |
| `source_refs` | Mandatory (≥1) | At least one Git path, issue URL, or PR URL |
| `evidence_refs` | Mandatory (≥1) | At least one `evidence_id` from `evidence_ref` table |
| `confidence` | Mandatory | In [0.0, 1.0] |
| `ttl` | Mandatory | Integer ≥ 0; 0 = no automatic expiry |
| `created_by` | Mandatory | Agent identifier of record creator |
| `created_at` | Mandatory | ISO 8601 datetime |
| `expires_at` | Mandatory if `ttl > 0` | ISO 8601 datetime; MUST be consistent with `created_at + ttl` |

Optional fields: `stale_after`, `superseded_by`, `comment`.

---

## 4. Validation Gates Before Import

The JSONL importer MUST enforce the following gates for `agent_memories.jsonl`:

| Gate | Severity | Description |
|---|---|---|
| Missing mandatory field | Blocking | Any field in Section 3 is absent |
| `scope` is empty, `"global"`, or `"*"` | Blocking | Unscoped global write is forbidden |
| `namespace` is empty | Blocking | Namespace is required |
| `content` is empty | Blocking | Empty memory is invalid |
| `source_refs` is empty | Blocking | No provenance = invalid |
| `evidence_refs` is empty | Blocking | No evidence = invalid |
| `confidence` outside [0.0, 1.0] | Blocking | Numeric contract violation |
| `ttl < 0` | Blocking | Negative TTL is invalid |
| `expires_at` absent when `ttl > 0` | Blocking | Expiry constraint must be explicit |
| `memory_type` not in canonical set | Blocking | Unknown type |
| `evidence_refs[*]` not in current batch | Warning | Cross-batch reference may be valid from prior import |
| `superseded_by` references non-existent ID in batch | Warning | Target may exist in DB from prior import |
| Circular `superseded_by` chain within batch | Blocking | Circular supersede is invalid |
| Memory content contains trading state | Blocking | G1 violation — no orders/positions/fills/balances |
| Memory content contains secrets | Blocking | G2 violation — no credentials/keys |
| Memory content claims LR-GO or Echtgeld-GO | Blocking | G3/G4 violation |

## 5. Stale and Supersede Handling at Import Time

### 5.1 Supersede at Import

- If a batch contains both an existing record being superseded and its replacement, the
  importer MUST process the replacement first (or in the same transaction), then mark the
  superseded record.
- The importer MUST NOT delete the superseded record; it remains in the audit trail with
  `superseded_by` set.
- Circular supersede chains within a batch are a blocking validation error.

### 5.2 Stale Detection at Import

- The importer SHOULD flag records where `expires_at` is in the past relative to the import
  timestamp as a warning (the record is being imported already stale).
- Importing already-stale records is not a blocking error — they may be imported for
  historical/audit purposes — but the finding MUST be surfaced.
- Stale memory records from prior batches are NOT automatically invalidated by a new import.

## 6. Relationship to `scoped-agent-memory-model-v1.md`

This document is a supplementary import policy. The canonical memory model definition
(types, TTL recommendations, write rules, unscoped-global rules, supersede semantics)
remains in `scoped-agent-memory-model-v1.md`.

In case of conflict, `scoped-agent-memory-model-v1.md` takes precedence on model semantics.
This document is authoritative only for import-specific validation gates and authoring paths.
