# Session Log: 2026-03-24 — Grafana DatasourceError Fix #1266 #1267

**Datum:** 2026-03-24
**Status:** REOPENED — Fix-Rollback (Grafana 11.4.7 Provisioning-Inkompatibilität)
**Issues:**
- #1266 — [ALERTING][GRAFANA] Orders-Rejected-Alert scheitert an Prometheus-Zielauflösung über DNS-Host `cdb_prometheus`
- #1267 — [ALERTING][GRAFANA] High-Error-Rate-Alert scheitert ebenfalls als DatasourceError wegen nicht auflösbarem Prometheus-Host `cdb_prometheus`

---

## Ausgangslage

Während des Soak-Runs wurden mehrere E-Mails mit `alertname=DatasourceError` zugestellt, die
wie echte Produktions-Incidents aussahen. Die Fehlermeldung:

```
failed to execute query [A]: Post "http://cdb_prometheus:9090/api/v1/query_range":
dial tcp: lookup cdb_prometheus on 127.0.0.11:53: no such host
```

Betroffen: `CDB - Orders Rejected` (#1266) und `CDB - High Error Rate (>5%)` (#1267).

---

## Root Cause

**Transiente DNS-Auflösung + `execErrState: Error`-Policy:**

1. **DNS-Auflösung:** Während des Environment-Interruption-Events (Bulk-Restart aller Container)
   entfernt Docker den DNS-Eintrag für `cdb_prometheus`, solange der Container gestoppt ist.
   `127.0.0.11` ist Docker's embedded DNS-Resolver. In der Stop-/Restart-Lücke schlägt
   `lookup cdb_prometheus` mit `no such host` fehl.

2. **Error-Policy:** Beide betroffenen Alert-Regeln hatten `execErrState: Error`. Grafana setzt
   bei Datasource-Evaluierungsfehler den Alert-State sofort auf `Error` und stellt via
   konfigurierter Mail-Contact-Point zu — sieht aus wie ein echtes High-Severity-Incident.

**Die Datasource-URL ist korrekt:** `http://cdb_prometheus:9090` ist der richtige Hostname,
weil beide Services (`cdb_grafana`, `cdb_prometheus`) in `compose.red.yml` auf `cdb_network` liegen
und der Service-Name dem Container-Name `cdb_prometheus` entspricht. Das Problem liegt
ausschließlich in der Error-Handling-Policy.

---

## Fix

```yaml
# Alt (erzeugt noisy DatasourceError-Mails während Prometheus-Restart):
execErrState: Error

# Neu (behält letzten bekannten State bei Datasource-Ausfall):
execErrState: KeepLastState
```

Geändert in:
- `infrastructure/monitoring/grafana/provisioning/alerting/orders_rejected.yml`
- `infrastructure/monitoring/grafana/provisioning/alerting/high_error_rate.yml`

`KeepLastState` bedeutet: Wenn Prometheus temporär nicht erreichbar ist, behalten die Alert-Regeln
ihren letzten Evaluierungszustand (typischerweise `Normal`/`OK`). Sobald Prometheus wieder verfügbar
ist, evaluieren die Regeln normal weiter. Echte Alerts feuern weiterhin korrekt.

`circuit_breaker.yml` wurde NICHT geändert — die Regel hatte einen anderen Fehler (#1265,
ungültiger Operator `gte`), der bereits repariert ist. `execErrState: Error` für den kritischen
Circuit-Breaker ist vertretbar.

## Restunsicherheiten

- Der Fix verhindert Mails bei TRANSIENTEM Datasource-Ausfall. Wenn Prometheus für längere
  Zeit ausfällt (nicht nur Restart-Fenster), sendet Grafana weiterhin keine Mails für diese
  Regeln — das Monitoring-Signal fehlt dann stärker. Für den aktuellen Kontext (Soak-Monitor
  mit bekanntem Restart-Risiko) ist das akzeptabel.
- `noDataState: NoData` bleibt unverändert. Wenn Prometheus verfügbar ist aber keine Daten
  zurückgibt (leere Series), setzt Grafana den State auf NoData. Das ist separates Verhalten
  und nicht Teil dieses Fixes.
- Ein echter, langfristiger Prometheus-Ausfall würde jetzt nicht durch diese Alert-Regeln
  gemeldet. Dafür wäre ein separater Prometheus-Health-Alert nötig (nicht Scope dieses Issues).

## Nachtrag 2026-03-24 — Fix-Rollback (PR #1273 operativer Cleanup)

**Problem beim Aktivieren:** `KeepLastState` wird von Grafana 11.4.7-ubuntu im
Provisioning-YAML-Parser **nicht akzeptiert**:
```
failure parsing rules: rule 'CDB - Orders Rejected' failed to parse:
unknown Error state option KeepLastState
```
Grafana startet nicht (Restart-Loop). Der Wert ist in der Grafana HTTP API als gültig
dokumentiert (`ExecutionErrorState` enum), aber der Provisioning-Pfad dieser Version
kennt ihn nicht.

**Aktion:** `execErrState: KeepLastState` → `execErrState: Error` zurückgerollt.
Commit: `77e4a79 revert(alerting): revert KeepLastState — not supported by Grafana 11.4.7-ubuntu`

**Issues:** #1266 und #1267 reopened. Root Cause korrekt diagnostiziert, Implementierung blockiert.

**Offene Lösungsoptionen (Entscheidungsvorlage):**

| Option | Aufwand | Risiko | Empfehlung |
|---|---|---|---|
| A) Grafana-Image-Upgrade auf kompatible Version (z.B. latest) | mittel | niedrig (Provisioning-YAML bleibt unverändert) | **Bevorzugt** — löst das Problem sauber |
| B) Alertmanager-Routing: separate Route für `alertname=DatasourceError` → dedizierter Receiver (z.B. silence/low-prio-mail) | mittel | niedrig | Alternative wenn Grafana-Version nicht angehoben werden soll |
| C) `execErrState: Alerting` statt `Error` | minimal | mittel (würde als normaler Alert feuern, nicht als DatasourceError — immer noch noisy) | Nicht empfohlen |

**Empfehlung:** Option A (Grafana-Version prüfen/hochstufen). Nach Upgrade kann der Fix aus PR #1273
(f22ad9f) direkt reaktiviert werden. Option B ist sinnvoll als unabhängige Verbesserung
(DatasourceError-Routing), ersetzt aber nicht Option A.
