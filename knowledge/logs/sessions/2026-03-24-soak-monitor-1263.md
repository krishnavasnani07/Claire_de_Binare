# Session Log: 2026-03-24 — Service Health Count Fix #1263

**Datum:** 2026-03-24
**Status:** CLOSED
**Issues:** #1263 — [SOAK][OBSERVABILITY] Service-Health-Monitor meldet inkonsistente Soll-/Ist-Zählung (`22/8 services running`)

---

## Ausgangslage

Check 2 in `soak_monitor.sh` meldete `✓ All 22/8 services running` — semantisch falsch
und vertrauensschädigend für Operator-Reviews.

---

## Root Cause

Zwei unabhängige Bugs kombinieren sich:

**Bug 1 — Veraltete Baseline:**
```bash
EXPECTED_SERVICES=8
```
Der Wert `8` entspricht den 8 BLUE-Core-Trading-Services ohne `cdb_postgres`, `cdb_redis`
und ohne die RED-Signal-Services `cdb_ws`, `cdb_signal`. Er wurde nie an die aktuelle
Stack-Realität angepasst.

**Bug 2 — Breiter Inventur-Filter:**
```bash
RUNNING_SERVICES=$(docker ps --filter "name=cdb_" --filter "status=running" | grep -c "cdb_" || true)
```
`grep -c "cdb_"` zählt ALLE Container, deren Name `cdb_` enthält — SUT-Services,
Observability-Sidecars (Prometheus, Grafana, Exporter), Infra-Helper (`cdb_surrealdb`,
`cdb_gh_runner`), weitere Service-Varianten (`cdb_market_eth`) gemeinsam. Im vollen
BLUE+RED+Exporter-Runtime ergibt das 22+.

Ergebnis: links Inventur-Zählung (22), rechts veraltete Baseline (8) → `22/8 services running`.

---

## Fix

Ersetze `EXPECTED_SERVICES=8` + breiten Filter durch eine explizite SUT-Service-Liste:

```bash
# ZRP-relevant SUT services: BLUE core + data layer + RED signal services.
SUT_SERVICES="cdb_postgres cdb_redis cdb_market cdb_candles cdb_regime cdb_allocation \
              cdb_risk cdb_execution cdb_db_writer cdb_paper_runner cdb_ws cdb_signal"
EXPECTED_SERVICES=12

_RUNNING_NAMES=$(docker ps --filter "status=running" --format "{{.Names}}" 2>/dev/null || true)

RUNNING_SERVICES=0
for _svc in $SUT_SERVICES; do
  if echo "$_RUNNING_NAMES" | grep -qx "$_svc"; then
    RUNNING_SERVICES=$((RUNNING_SERVICES + 1))
  else
    MISSING_SVC_LIST="$MISSING_SVC_LIST $_svc"
  fi
done

TOTAL_CDB=$(echo "$_RUNNING_NAMES" | grep -c "^cdb_" || echo 0)
# Ausgabe: "12/12 SUT services running (inventory: 22 cdb_* containers)"
```

**SUT-Set Herleitung:**
- `cdb_postgres`, `cdb_redis` — BLUE data layer, ZRP-kritisch
- `cdb_market`, `cdb_candles`, `cdb_regime`, `cdb_allocation`, `cdb_risk`, `cdb_execution`, `cdb_db_writer`, `cdb_paper_runner` — BLUE core (8 Services)
- `cdb_ws`, `cdb_signal` — RED signal services, ZRP-relevant

Nicht enthalten (bewusst):
- Observability: `cdb_prometheus`, `cdb_grafana`, `cdb_alertmanager`, Exporter
- Infra: `cdb_gh_runner`, `cdb_surrealdb`, `cdb_reports`

## Restunsicherheiten

- `cdb_node_exporter` ist nicht in compose.red.yml, taucht aber laut Issue-Artefakt auf.
  Das bedeutet, es wird von einem anderen Compose-File gestartet und ist korrekt
  NICHT im SUT-Set — kein Problem für den Fix.
- `cdb_market_eth` (ETHUSDT-Service) ist bewusst nicht im SUT-Set (es ist eine optionale
  Erweiterung, kein Core-ZRP-Service). Sollte `cdb_market_eth` Teil des ZRP werden,
  muss `SUT_SERVICES` und `EXPECTED_SERVICES` angepasst werden.
- Der breite cdb_*-Inventur-Count wird jetzt nur noch informativ ausgegeben.
  Wenn Grafana/Prometheus ausfallen, wird das NICHT als Health-Warning gemeldet.
  Das ist absichtlich, könnte aber überraschen, wenn jemand den Monitor als
  Gesamt-Stack-Indicator betrachtet.
