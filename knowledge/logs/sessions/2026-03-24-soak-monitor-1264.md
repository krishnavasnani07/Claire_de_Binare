# Session Log: 2026-03-24 — Disk-Space-Evidence Fix #1264

**Datum:** 2026-03-24
**Status:** CLOSED
**Issues:** #1264 — [SOAK][OBSERVABILITY] Disk-Space-Check liefert keine verwertbare Evidenz

---

## Ausgangslage

Check 5 in `soak_monitor.sh` meldete bei jedem Checkpoint `⚠️  Could not determine disk usage`.
Keine Disk-Evidenz in den Artefakten außer im (nie eingetretenen) >90%-Critical-Fall.

---

## Root Cause

**Namespace-/Runtime-Mismatch + pipefail-Interaktion:**

```bash
DISK_USAGE=$(df -h /var/lib/docker 2>/dev/null | awk 'NR==2 {print $5}' | sed 's/%//' || echo "unknown")
```

1. `/var/lib/docker` existiert nur auf der Docker-Host-Seite (Linux-Daemon-Kontext).
   Im `lr040_soak_monitor`-Container (ubuntu:22.04) ist nur `/repo` und `/var/run/docker.sock`
   gemountet — `/var/lib/docker` ist `No such file or directory`.

2. `set -euo pipefail` wurde zu Beginn gesetzt. `set +e` entfernt nur `-e` (errexit),
   aber NICHT `-o pipefail`. Mit aktivem `pipefail` bekommt eine Pipeline den Exit-Code
   des rechtesten fehlgeschlagenen Befehls. Da `df` (Exit 1) scheitert, schlägt die
   gesamte Pipeline fehl → `|| echo "unknown"` greift → `DISK_USAGE="unknown"`.

3. Selbst wenn ein Wert ermittelt würde: kein persistiertes Disk-Artefakt außer im
   Critical-Fall (>90%) — Evidenzkette hat eine strukturelle Lücke.

---

## Fix

Ersetze den nicht-erreichbaren `/var/lib/docker`-Pfad durch zwei Container-erreichbare Quellen:

1. **`df /repo`** — immer gemountet (Cron: `cd /repo`), gibt Usage der Partition
   auf der Artefakte geschrieben werden
2. **`docker system df`** — Docker-Speicher (Images, Container, Volumes, Build Cache)
   via Socket, ohne Host-Pfad-Zugriff

Zusätzlich: `disk_evidence_<H>h.txt`-Artefakt bei **jedem** Checkpoint schreiben,
nicht nur bei Critical.

```bash
ARTIFACT_DISK_PCT=$(df /repo 2>/dev/null | awk 'NR==2 {print $5}' | sed 's/%//' || true)
ARTIFACT_DISK_FREE=$(df -h /repo 2>/dev/null | awk 'NR==2 {print $4}' || true)
DOCKER_DF_OUT=$(docker system df 2>/dev/null || echo "  [docker system df not available]")
# → schreibt disk_evidence_<H>h.txt bei jedem Checkpoint
```

## Restunsicherheiten

- Auf WSL2/Docker Desktop zeigt `df /repo` die WSL2-VHD-Partition (nicht Windows C:).
  Wenn Docker Desktop die Logs/Images auf dem WSL2-VHD speichert, ist das tatsächlich
  die relevante Partition. Wenn kritische Daten auf einem anderen Windows-Laufwerk liegen,
  würde das nicht erkannt. Für den aktuellen Soak-Kontext ist das akzeptabel.
- `docker system df` zeigt Docker-interne Space-Nutzung, nicht Host-Filesystem-%.
  Das ergänzt die Disk-Evidence, ist aber kein direkter Ersatz für `df`-Host-Percentages.
- Der `lr040_soak_gate_eval.py` wertet Disk-Evidence noch nicht automatisch aus.
  Die neuen `disk_evidence_<H>h.txt`-Dateien sind für Operator-Review verfügbar.
