**Datum:** 2026-03-24

---

## Ausgangslage

Das Issue #1268 wurde während des aktiven Soak-Runs `artifacts/soak_test_20260324_000002` gemeldet.
Beobachtete Fehlermeldungen:

```
./infrastructure/scripts/soak_monitor.sh: line 128: 08: value too great for base (error token is "08")
./infrastructure/scripts/soak_monitor.sh: line 134: 09: value too great for base (error token is "09")
```

Betroffen waren Check 3 (Resource Snapshot, 6h-Intervall) und Check 4 (DB Growth, 12h-Intervall).
Der Monitor brach die Checks nicht ab, übersprang aber die Verzweigung für Checks 3/4 still.

---

## Root Cause

Die **alte** Version des Skripts setzte:

```bash
HOUR=$(date +%H)   # Lokale Zeit, kein -u-Flag → z.B. "08" in MESZ um 07:00 UTC
```

Anschließend:

```bash
if [ "$((HOUR % 6))" -eq 0 ]; then    # Bash: $(( 08 % 6 )) → oktal → FEHLER
if [ "$((HOUR % 12))" -eq 0 ]; then   # Bash: $(( 09 % 12 )) → oktal → FEHLER
```

Bash-Arithmetik interpretiert `08` und `09` als Oktalzahlen (die ungültig sind).
Das Skript lief mit `set +e` weiter, aber die `if`-Bedingungen wurden nicht ausgewertet —
Checks 3 und 4 wurden in den betroffenen Stunden still übersprungen.

In Deutschland/MESZ (UTC+1) trifft dies bei UTC 07:00 (→ lokal 08:00) und UTC 08:00 (→ lokal 09:00) zu.

---

## Fix

Der Fix erfolgte **als Nebeneffekt von #1271**: Das Skript wurde refactored, um statt
`HOUR=$(date +%H)` die variable `ELAPSED_HOURS` zu verwenden:

```bash
RUN_START_EPOCH=$(cat "$RUN_START_FILE")
NOW_EPOCH=$(date -u +%s)
ELAPSED_HOURS=$(( (NOW_EPOCH - RUN_START_EPOCH) / 3600 ))   # immer Plain-Integer
```

Epoch-Arithmetik erzeugt niemals führende Nullen. `$((8 % 6))` = 2 und `$((9 % 6))` = 3
funktionieren korrekt.

**Das Issue #1268 war durch #1271 inhaltlich bereits behoben.** Fehlende explizite
Regressionstests waren der einzige ausstehende Teil.

## Restunsicherheiten

- Der Fix (via #1271) basiert auf Python-seitigen Hilfsfunktionen, die die Bash-Logik
  spiegeln. Direkte Bash-Unit-Tests existieren nicht. Sollte das Bash-Skript künftig
  auf `date +%H`-artige Konstrukte zurückgeändert werden, greift kein direkter Bash-Test.
  → Mitigiert durch klare Docstrings + Commit-Message, die auf #1268 verweisen.

- `last_checkpoint.txt` könnte theoretisch von einer alten Skript-Version mit Zero-Padded
  Werten (`08`) befüllt sein. Der `-le`-Vergleich in `test` (nicht Arithmetik-Expansion)
  ist oktal-sicher, daher kein aktives Risiko.
